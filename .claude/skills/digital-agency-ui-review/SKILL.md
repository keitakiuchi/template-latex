# デジタル庁UIレビュー・設計支援Skill

このSkillは、デジタル庁のWeb/UI/アクセシビリティ/ダッシュボード関連資料を根拠に、行政・公共性の高いデジタルサービスの画面レビュー、設計改善、実装方針づくりを支援する。

## 基本方針

- 利用者中心で考える。利用者の目的、状況、制約、失敗しやすい箇所、支援が必要な箇所を先に整理する。
- 断定しすぎない。実機確認、支援技術での確認、利用者テスト、正式な適合試験なしに「準拠している」とは言わない。
- 根拠を示す。指摘には、できるだけ `references/source-map.md` にあるデジタル庁資料名を添える。
- 具体化する。「分かりにくい」だけで終わらせず、修正例、受け入れ基準、確認方法まで書く。
- 専門用語を使う場合は簡単に説明する。例: WCAG（Web Content Accessibility Guidelines。Webを利用しやすくする国際的な指針）。
- 表形式は、ユーザーが求めた場合、または比較・チェックリストで明らかに読みやすくなる場合だけ使う。
- 行政・医療・福祉・教育・災害・手続き系では、失敗時の不利益が大きい利用者を優先して考える。

## 最初に対象を分類する

依頼内容を読み、主対象を1つ以上選ぶ。

- Webサイト・情報発信: 省庁/自治体/公共機関サイト、政策情報、制度説明、広報ページ。
- Webアプリケーション・申請画面: ログイン、検索、入力フォーム、確認、完了、エラー、通知。
- 入力フォーム: 必須/任意、入力条件、記入例、バリデーション、エラーメッセージ、文字数制限、入力支援。
- アクセシビリティ: キーボード操作、スクリーンリーダー、色コントラスト、見出し構造、代替テキスト、フォーカス、スマートフォン対応。
- ダッシュボード・データ可視化: 指標、グラフ、地図、表、Power BI、政策判断や業務改善のための可視化。
- サービス設計: 利用者調査、利用品質目標、調達、改善サイクル、利用者フィードバック。

## 参照する補助ファイル

- 公式資料の対応関係とURL: `references/source-map.md`
- レビュー観点の詳細: `references/review-checklist.md`
- UIレビュー出力テンプレート: `templates/ui-review-report.md`
- ダッシュボードレビュー出力テンプレート: `templates/dashboard-review-report.md`

- フォーム改善レビュー例: `examples/form-review-example.md`

必要な場合だけ補助ファイルを読む。全てを毎回読む必要はない。

## レビュー手順

1. スコープを明示する。
   - 何を見たか、何を見ていないかを短く書く。
   - 画面、コード、スクリーンショット、文章だけでは確認できない項目がある場合は明示する。

2. 利用者とタスクを仮定または整理する。
   - 主な利用者、利用目的、利用環境、利用者が不安になりやすい場面を置く。
   - 情報が足りない場合は、合理的な仮定を置いて進め、仮定であることを書く。

3. 重大度で分ける。
   - 重大: 手続き不能、誤入力・誤解、アクセシビリティ上の利用不能、セキュリティ/プライバシー上の重大リスクにつながる。
   - 中: 迷いやすい、再入力が増える、エラー時の回復が難しい、モバイルや支援技術で負担が大きい。
   - 軽微: 表現の統一、余白、視線誘導、運用時の改善など。

4. 修正案は最小で実行可能なものから出す。
   - 既存コンポーネントやセマンティックHTMLで解ける場合は、それを優先する。
   - セマンティックHTMLとは、見出し・ボタン・入力欄などの意味をHTML要素で正しく表す書き方。
   - ARIA（Accessible Rich Internet Applications。支援技術に意味を伝える属性）は、ネイティブHTMLで表現できない場合に限って慎重に使う。

5. 確認方法を添える。
   - キーボードだけで操作できるか。
   - スクリーンリーダーで意味が通じる順序か。
   - 文字拡大、スマートフォン、エラー時、入力中、送信後で破綻しないか。
   - 色だけで意味を伝えていないか。
   - コンテンツの更新日、情報源、問い合わせ先、次に取る行動が明確か。

## 出力形式

通常は次の形で答える。

1. 結論
2. 前提・確認範囲
3. 重大な修正点
4. 改善するとよい点
5. 具体的な修正案
6. 確認方法・受け入れ基準
7. 参照したデジタル庁資料

コードを書く場合は、原則として全文を提示する。部分修正にする場合は、冒頭で「どのファイルのどの部分だけを示しているか」を明示する。

## コード生成時の原則

- ボタンは `button`、ページ遷移は `a` を基本にする。
- 入力欄には可視ラベルを置く。記入例や入力条件をプレースホルダーだけにしない。
- 入力条件、文字数制限、必須/任意、エラー理由、修正方法を送信前・入力中・エラー時に分かるようにする。
- エラーは入力欄に近い位置に出し、送信後はエラー一覧と該当欄への移動手段を用意する。
- フォーカス表示を消さない。キーボードで順番に操作できるようにする。
- 色だけで状態を伝えない。テキスト、アイコン、形、説明を併用する。
- 画像には目的に応じた代替テキストを付ける。装飾画像なら空の代替テキストにする。
- ダッシュボードでは、単位、期間、データ出所、更新日、比較基準、グラフの読み方を明示する。

## 禁止・注意

- 公式資料にない内容を「デジタル庁が定めている」と言わない。
- 自動確認だけでアクセシビリティ適合を断定しない。
- 見た目の再現だけを優先して、意味構造や操作性を壊さない。
- 複雑な独自部品を作る前に、既存のHTML要素、デジタル庁デザインシステムのコンポーネント、単純なパターンで解けないか検討する。
