# フォーム改善レビュー例

## 対象

住所入力フォームの「番地・建物名」欄。

## 問題

記入例がプレースホルダーだけで表示されている。入力を始めると例が消えるため、利用者は入力中に形式を確認できない。エラーが出た場合も、入力値と正しい例を比較しにくい。スクリーンリーダー利用者にも、入力前に十分な説明が伝わらない可能性がある。

## 修正案

- 入力欄の前または直下に、常に見える説明文として記入例を置く。
- ラベルは省略しない。
- 入力条件がある場合は、文字数、使用できる文字、例外を明記する。
- プレースホルダーは補助的に使うだけにする。

## 修正例

```html
<label for="address-line2">番地・建物名</label>
<p id="address-line2-help">例: 1-2-3 デジタル庁マンション101号室。建物名がない場合は番地まで入力してください。</p>
<input
  id="address-line2"
  name="addressLine2"
  type="text"
  autocomplete="address-line2"
  aria-describedby="address-line2-help"
/>
```

## 受け入れ基準

- 入力前、入力中、入力後のいずれでも記入例を確認できる。
- スクリーンリーダーでラベルと説明が入力欄に関連付いて読み上げられる。
- エラー時に、どの条件を満たしていないか、どう直せばよいかが分かる。

## 根拠資料

- 政府情報システムのUI改善: プレースホルダー依存を避け、記入例を常に確認できる形にする改善例。
- DS-670.1 ユーザビリティガイドライン / UIチェックリスト: 入力フォームにおけるバリデーションと入力支援。
- DS-671.2 ウェブアクセシビリティ導入ガイドブック: スクリーンリーダー等への配慮。
