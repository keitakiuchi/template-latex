# AGENTS.md

このファイルは、Codex を含む AGENTS.md 対応エージェント向けのプロジェクト共有指示です。
長い一般論ではなく、このリポジトリ固有で毎回読む価値がある内容だけを書きます。

## 応答言語

- ユーザー向けの回答、進捗報告、要約は常に日本語で行う。

## このリポジトリの性格

- このリポジトリ自体が AI 開発運用のテンプレートであり、現時点で `src/` や `tests/` が未作成でも不整合ではない。
- 新しくアプリケーションコードやサンプルコードを追加する場合は、ルート直下ではなく `src/` または `app/` に置く。
- テストは `tests/`、設定は `config/`、補助スクリプトは `utility/` に置く。
- `workbench/` は一時的な実験、`analysis_steps/` は再現可能な分析手順、`results/` は保存すべき成果物に使う。

## 実行前提

- プロジェクトコードを実行する前に、必ず `python utility/check_env.py` を実行する。
- 仮想環境名を `myenv` や `py-review` に決め打ちしない。最新の手順は `README.md` と `environment.yml` を正本として扱う。
- ローカル開発の依存関係の正本は `environment.yml`。`requirements.txt` は主に GitHub Actions や互換用途として扱う。
- WSL で外付け SSD を扱う手順は `README.md` の `SSD_DRIVE` / `SSD_MOUNT` / `TEXLIVE_PATH` を正本とし、`D:` 固定ではなく `D:` / `E:` の両方を前提にする。

## 作業方針

- 可能な範囲で TDD を使い、まず失敗する最小のテストを書き、その後に最小修正を入れる。
- 変更は小さく保ち、確認は最小の対象から始める。
- 重い処理は勝手に端折って実行しない。実行負荷が高い場合は、必要なコマンドや手順を示したうえでユーザーに実行を依頼する。
- 時間のかかる処理は、進行状況が分かるようにコンソールへ進捗を表示する実装を優先する。
- 重いデータ解析を行う際には、チェックポイントを入れるなど、途中で処理が止まっても再開できるような対策を講じる。
- 特定のパッケージや依存関係が不足しているときは、その使用を避けて別の方法へ勝手に切り替えない。まず `environment.yml`、`README.md`、当該ツールの標準手順に沿ってインストールを試みる。権限、ネットワーク、実行環境の制約などで自分でインストールできない場合は、必要なパッケージ名と推奨コマンドを示してユーザーにインストールを依頼する。別実装や別パッケージで進めるのは、ユーザーの明示的な許可を得た後に限る。
- warning は勝手に抑制しない。抑制するときは許可を取る。
- フォールバックを勝手に入れない。入れるときは許可を取る。
- LLM 関連の実装では、古典的 NLP によるルールベースの実装を勝手に入れない。入れるときは許可を取る。
- 統計解析の処理に時間がかかる場合でも、勝手に設定を軽くして回避しない。サンプル数、反復回数、探索範囲、精度などを下げる前に、まずユーザーへ確認する。
- 新しいファイルやディレクトリを増やす前に、既存の構成と役割に合っているかを確認する。
- リポジトリ直下にはアプリ本体の実装やテスト本体を増やさない。

## デザイン規約

- UI、ダッシュボード、チャート、表、図、paper-ready figure を作成・修正するときは、まず `DESIGN.md` を読む。
- `DESIGN.md` は Google DESIGN.md 互換の YAML front matter と Markdown 本文を持つ共通デザイン規約として扱う。
- YAML front matter の design tokens は正確な値の正本、Markdown 本文は適用判断と例外判断の正本として扱う。
- Gemini だけでなく、Claude Code、Codex、Cursor でも同じ `DESIGN.md` を参照し、ツール固有の見た目指示を重複して増やさない。

## リサーチ方針

- 最新情報や変化しやすい情報を調べる場合は、一次情報と公式ドキュメントを優先する。
- Codex セッションでは組み込みの Web 検索を使ってよいが、結果には出典 URL を付ける。
- Codex 以外のツールで Web 検索が必要な場合は、原則として `gemini --prompt "WebSearch: <query>"` の形式を使う。

## セキュリティ

- `.env`、認証情報、秘密鍵、トークンを出力・要約・コミットしない。
- 破壊的操作、依存関係追加、CI/設定変更、外部ネットワーク通信は高リスクとして扱う。
- 外部入力やシェル引数は安全に扱い、危険なワンライナーを避ける。

## サブエージェント運用（「サブちゃん」トリガー）

- ユーザーが Codex に「サブちゃん」と伝えた場合、それだけで「サブエージェントを使って実行してほしい」という明示的な指示が与えられたものとして扱う。
- これは単なる使用許可ではなく、可能な範囲で `spawn_agent` を使ってサブエージェント実行を行うトリガーとして解釈する。
- メインエージェントは、破壊的操作や高リスク作業で別途承認が必要な場合を除き、追加確認なしでサブエージェントを起動して実行し、完了後に結果を要約して統合する。
- 例: 「サブちゃん、テストを実行して」「サブちゃんでリサーチして要約して」。
- 任意で「サブちゃん <名前>: <タスク>」形式のニックネーム指定を受け付ける。
- 破壊的操作や高リスク作業は既存の承認フロー・安全基準に従う。
